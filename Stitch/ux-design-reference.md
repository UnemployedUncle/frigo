# Frigo UX Design Reference

기준일: 2026-03-20

## 1. 문서 목적

이 문서는 현재 구현된 Frigo를 바탕으로 UX 디자인에 필요한 입력값을 정리한 문서입니다.

- 어떤 화면을 설계해야 하는지
- 각 화면에서 반드시 보여야 하는 정보가 무엇인지
- 어떤 액션과 상태 변화를 고려해야 하는지
- 현재 구현과 향후 개선 포인트를 어디까지 반영해야 하는지

## 2. 제품 UX의 기본 방향

Frigo의 UX는 아래 원칙을 유지해야 합니다.

- `text-first`: 사진보다 텍스트 정보 구조가 우선입니다.
- `lightweight`: 첫 화면 진입과 다음 행동이 빠르게 보여야 합니다.
- `operational`: 탐색보다 실행 중심입니다. 사용자는 바로 입력하고, 고르고, 조리해야 합니다.
- `low-friction`: 저장, 추천, 조리 시작 같은 핵심 액션은 짧은 경로여야 합니다.
- `browser-first cooking`: 조리 중에는 서버 왕복보다 현재 step 집중이 중요합니다.

## 3. 핵심 사용자 목표

디자인은 아래 사용자 목표를 빠르게 달성하도록 설계해야 합니다.

1. 지금 냉장고에 뭐가 있는지 빠르게 정리한다.
2. 가지고 있는 재료로 만들 만한 레시피를 빠르게 찾는다.
3. 부족한 재료를 바로 확인한다.
4. 조리 중에는 현재 해야 할 행동 하나에 집중한다.
5. 마음에 든 레시피와 실제로 해본 레시피를 구분해 남긴다.

## 4. 전체 IA와 주요 흐름

주요 화면:

1. Home
2. Fridge
3. Fridge Recommendations
4. Recipe Detail
5. Cook Mode

핵심 사용자 흐름:

1. `Home -> Recipe Detail -> Cook Mode -> Complete`
2. `Home -> Fridge -> Fridge Recommendations -> Recipe Detail`
3. `Fridge -> Recipe Detail -> Save`
4. `Recipe Detail -> Save`

추천 설계 포인트:

- Home은 탐색 시작점
- Fridge는 입력/관리 허브
- Recipe Detail은 조리 전 의사결정 화면
- Cook Mode는 실행 전용 화면

## 5. 공통 UX 시스템

### 5.1 공통 정보 계층

대부분의 화면은 아래 순서를 유지하는 편이 좋습니다.

1. 화면 목적
2. 핵심 액션
3. 주요 정보 블록
4. 보조 정보

### 5.2 공통 컴포넌트 후보

화면 전반에서 반복되는 구성 요소:

- 레시피 카드
- 저장 토글 버튼
- ingredient pill
- section panel
- 상태 배지 또는 freshness indicator
- step list
- timer readout
- empty state 블록
- error/message banner

### 5.3 공통 상태 표현

반드시 디자인해야 할 상태:

- 기본 상태
- 비어 있음
- 성공 상태
- 선택 오류
- 추천 결과 없음
- 저장됨 / 저장 안 됨
- 조리 중 / 일시정지 / 완료 직전

## 6. 화면별 디자인 요구사항

### 6.1 Home

화면 목적:

- 서비스의 현재 상태를 빠르게 파악
- 추천 레시피로 진입
- Fridge로 이동

반드시 보여야 하는 정보:

- Saved Recipes 수
- Completed Recipes 수
- 최근 완료 분포
- 추천 레시피 목록

주요 액션:

- `Open Fridge`
- `View Recipe`
- `Save/Unsave`

디자인 시 고려사항:

- Home은 대시보드처럼 보여야 하지만 복잡하면 안 됩니다.
- 저장 수와 완료 수는 한눈에 읽혀야 합니다.
- Cooking Grass는 시각적 재미보다 “최근 활동성” 전달이 우선입니다.
- 추천 카드는 한 번에 스캔 가능해야 합니다.

필수 상태:

- 추천 결과 존재
- 추천 결과 없음
- 저장된 카드
- 저장되지 않은 카드

보완 권장:

- 저장/완료 수와 추천 사이의 정보 위계를 더 분명히 정의
- 카드 hover/focus/pressed 상태 설계
- 잔디의 색상 단계 또는 밀도 단계 정의

### 6.2 Fridge

화면 목적:

- 재료 입력
- 재료 확인
- 재료 정리
- 추천용 재료 선택

반드시 보여야 하는 정보:

- 자연어 입력 영역
- 입력 예시 또는 가이드
- parsed preview
- 현재 냉장고 보유 목록
- 추천용 선택 영역
- 편집 가능 테이블

주요 액션:

- `Parse And Store`
- 재료 수정 저장
- 재료 삭제
- 추천용 재료 선택
- `Recommend From Selected Ingredients`

디자인 시 고려사항:

- 이 화면은 기능 밀도가 가장 높습니다.
- 입력, 확인, 편집, 추천 선택이 한 화면에 모여 있으므로 영역 분리가 중요합니다.
- 사용자에게 “지금 어디서 무엇을 해야 하는지”가 즉시 보여야 합니다.
- Text Fridge는 실제 냉장고 메타포를 줄 수 있지만, 편집 정확도는 테이블이 담당합니다.

필수 상태:

- 냉장고 비어 있음
- parse 성공 후 preview 표시
- 추천 선택 오류
- 항목 다수 보유 상태
- 유통기한 없음
- 임박 재료 존재

보완 권장:

- 선택 개수 카운터와 제한 룰을 실시간으로 표시
- `Text Fridge`와 `Edit Table`의 역할 구분을 더 명확히 설명
- 삭제 전 확인 여부 결정
- parse 결과를 저장 전 검토형으로 바꿀지 여부 검토

### 6.3 Fridge Recommendations

화면 목적:

- 사용자가 고른 재료 기준의 추천 결과 비교

반드시 보여야 하는 정보:

- 어떤 재료를 기준으로 추천했는지
- 추천 결과 목록
- 저장 상태

주요 액션:

- `Back To Fridge`
- `Home`
- `View Recipe`
- `Save/Unsave`

디자인 시 고려사항:

- Home 추천과 구분되는 맥락이 명확해야 합니다.
- 선택 재료 요약은 결과의 “이유”로 읽혀야 합니다.
- 결과가 적거나 없을 때도 다음 행동이 분명해야 합니다.

필수 상태:

- 결과 있음
- 결과 없음
- 저장됨 / 저장 안 됨

보완 권장:

- 선택 재료 수정 진입을 더 짧게 만들지 검토
- 추천 결과가 적을 때 재시도 액션 또는 안내 문구 강화

### 6.4 Recipe Detail

화면 목적:

- 이 레시피를 지금 할지 결정
- 부족 재료 확인
- 조리 플로우 이해
- 조리 시작

반드시 보여야 하는 정보:

- 레시피 제목, 요약
- 핵심 메타데이터
- 전체 재료
- shopping list
- 전체 workflow
- 총 조리 시간

주요 액션:

- `Save/Unsave`
- `Start Cooking`
- `Home`
- `Fridge`

디자인 시 고려사항:

- 정보량이 많기 때문에 스캔 순서가 중요합니다.
- `Start Cooking`은 최우선 CTA여야 합니다.
- shopping list는 재료 목록과 혼동되지 않아야 합니다.
- workflow는 조리 전 예습용으로 읽히면 됩니다.

필수 상태:

- 장보기 항목 있음
- 장보기 항목 없음
- 저장됨 / 저장 안 됨

보완 권장:

- 재료 목록과 shopping list를 시각적으로 더 강하게 분리
- 총 시간, 단계 수, 인분 등 요약 정보 묶음 정리

### 6.5 Cook Mode

화면 목적:

- 사용자가 현재 해야 할 한 단계에 집중하도록 지원

반드시 보여야 하는 정보:

- 현재 step 번호
- 현재 step 제목
- 현재 step 설명
- 현재 step 남은 시간
- 전체 남은 시간
- tool
- 필요한 재료
- 전체 workflow 내 현재 위치

주요 액션:

- `Pause`
- `Resume`
- `Stop`
- `Complete Now`

디자인 시 고려사항:

- 이 화면은 정보량보다 집중도가 우선입니다.
- 현재 step은 가장 크게 보여야 합니다.
- 전체 workflow는 보조 내비게이션 역할이면 충분합니다.
- 시간이 줄어드는 감각이 직관적으로 전달되어야 합니다.
- `Stop`과 `Complete Now`는 의미가 다르므로 위험도 구분이 필요합니다.

필수 상태:

- 진행 중
- 일시정지
- 마지막 step
- 자동 완료 직전

보완 권장:

- step 전환 순간의 피드백 설계
- 완료 직후 성공 상태 화면 또는 전환 피드백 검토
- 모바일에서 타이머와 액션 버튼 고정 방식 검토

## 7. UX에서 반드시 정의해야 할 상태/예외

디자인 산출물에는 아래 예외 상태가 포함되어야 합니다.

- 냉장고가 비어 있을 때 Fridge 화면
- 잘못된 재료 개수 선택 시 Fridge 에러 상태
- 추천 결과가 0개일 때 Fridge Recommendations
- shopping list가 비어 있을 때 Recipe Detail
- 저장 토글 직후 상태 변화
- Cook Mode에서 Pause 후 Resume
- Cook Mode에서 Stop 선택 시 이탈 판단
- Cook Mode 자동 완료 직후 전환

## 8. 인터랙션 원칙

현재 제품 성격상 아래 인터랙션 원칙이 적합합니다.

- 핵심 액션은 항상 1차 버튼으로 분명하게 드러나야 합니다.
- 보조 액션은 과도하게 많아 보이지 않아야 합니다.
- 저장, 삭제, 중단처럼 의미가 다른 액션은 색/강조 체계가 달라야 합니다.
- 추천과 조리 시작은 “다음 단계로 이동”하는 감각이 분명해야 합니다.
- 조리 화면은 읽기보다 실행에 맞춘 큰 타이포와 큰 터치 타겟이 필요합니다.

## 9. 콘텐츠/카피 원칙

카피는 아래 기준을 권장합니다.

- 짧고 명령형
- 설명은 보조, 액션은 직접적
- 냉장고/조리/저장 상태를 즉시 이해할 수 있는 단어 사용
- 기술 용어보다 생활 언어 우선

예시 톤:

- `냉장고 재료를 입력하세요`
- `선택한 재료로 추천받기`
- `부족한 재료`
- `지금 단계`
- `조리 완료`

## 10. 반응형과 디바이스 고려사항

모바일 기준으로 특히 검토해야 할 점:

- Fridge의 고밀도 영역 분리
- 선택 카드와 편집 폼의 세로 스택 구조
- Recipe Detail의 긴 정보 스크롤 정리
- Cook Mode의 타이머 가시성 유지
- 조리 중 버튼이 화면 아래로 밀리지 않도록 하는 구조

## 11. 현재 디자인 작업에서 제외해도 되는 범위

이번 UX 설계에서 깊게 다루지 않아도 되는 범위:

- 사진 업로드 UX
- 개인화 온보딩
- 주간 식단 캘린더
- 세션 재개 경험
- 소셜 기능
- 복수 사용자/권한 구조

## 12. 디자이너가 우선 결정해야 할 항목

우선순위가 높은 결정 항목:

1. Home의 정보 위계와 첫 CTA 구조
2. Fridge 화면의 4개 영역 분리 방식
3. 추천 선택 개수 제약을 어떻게 시각적으로 보여줄지
4. Recipe Detail의 정보 스캔 순서
5. Cook Mode의 집중형 레이아웃과 위험 액션 구분
6. 저장 상태와 완료 상태를 어떤 시각 언어로 구분할지

## 13. 권장 산출물

디자인 작업 산출물은 아래 순서를 권장합니다.

1. 전체 사용자 흐름 다이어그램
2. 화면별 lo-fi wireframe
3. 상태/예외 포함 hi-fi 시안
4. 공통 컴포넌트와 상태 정의
5. 모바일 변형안
