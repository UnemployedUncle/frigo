# 요리 여정 핵심 MVP PRD

## 1. 목표

이 문서는 요리 여정의 첫 MVP에서 실제로 개발할 핵심 기능만 확정하기 위한 PRD입니다. 이번 MVP는 냉장고 재료를 정리하고, 유통기한이 임박한 재료를 우선 사용하는 레시피를 추천하고, 필요한 장보기 목록을 만들고, 선택한 레시피를 단계별 workflow로 따라갈 수 있게 만드는 데 집중합니다.

냉장고 정리와 레시피 검색 계획 수립에는 agent 방식을 도입하되, 저장 구조는 고정된 데이터 스키마를 사용해 실제 CRUD가 가능하도록 설계합니다.

## 2. 이번 MVP에서 개발할 기능

- 냉장고 관리 기능
- 레시피 추천 기능
- 장보기 목록 기능
- 레시피 workflow 전환 기능

## 3. 이번 MVP에서 개발하지 않는 기능

- 홈 대시보드
- 사용자 계정 및 로그인
- 주간 식단 추천
- 주간 장보기 최적화
- 개인화 추천 고도화
- 레시피 수정 및 사용자 레시피 저장
- 타이머, 음성 안내, 스와이프 제스처
- 가격 정보 연동

## 4. 핵심 제품 결정

### 4.1 냉장고 관리

- 입력 방식은 `텍스트 기반 자연어 입력`만 지원합니다.
- 사용자는 한 개의 입력창에 재료 정보를 자연어로 입력합니다.
- 입력 예시는 `시금치 1봉지 3월 18일까지, 새우 200g 3월 17일까지, 버터 1개` 형태를 기준으로 합니다.
- 냉장고 관리 agent는 OpenRouter API를 호출해 입력 내용을 고정된 스키마의 재료 데이터로 파싱합니다.
- 파싱 결과는 고정된 DB 스키마에 맞춰 저장되어야 합니다.
- 표의 컬럼은 `id`, `재료명`, `정규화 재료명`, `수량`, `단위`, `유통기한`, `남은 일수`, `생성일`, `수정일`로 고정합니다.
- 사용자는 표에서 각 행을 직접 추가, 수정, 삭제할 수 있어야 합니다.
- 유통기한이 없는 재료는 빈 값으로 저장하되 추천 우선순위는 낮게 처리합니다.
- 동일 재료가 여러 번 들어오면 MVP에서는 자동 병합하지 않고 별도 행으로 저장합니다.
- OpenRouter 응답이 스키마에 맞지 않으면 저장 전에 서버에서 검증하고 보정해야 합니다.

### 4.2 레시피 추천

- 추천 소스는 외부 API가 아닌 `내부 레시피 데이터베이스`입니다.
- 레시피 추천은 냉장고 전체 재료를 그대로 검색에 사용하는 대신, agent가 검색용 주재료를 먼저 추출한 뒤 계획적으로 검색 범위를 넓혀가는 방식으로 구성합니다.
- 기본 설정은 `주재료 5개`를 선택해 첫 검색을 시작하는 것입니다.
- 검색 결과가 충분하지 않으면 주재료 수를 `5 -> 4 -> 3 -> 2 -> 1` 순서로 줄이며 검색 범위를 넓힙니다.
- 주재료 선택 우선순위는 아래 순서로 고정합니다.
- 1순위: 유통기한이 임박한 재료
- 2순위: 수량이 많은 재료
- 3순위: 레시피 주재료로 쓰일 가능성이 높은 재료
- agent는 각 검색 단계마다 `선택 재료`, `검색 쿼리`, `검색 이유`, `다음 단계 조건`을 포함한 search plan을 세워 차례대로 실행해야 합니다.
- 추천 결과는 각 단계 검색 결과를 합친 뒤 중복 제거하여 최대 5개까지 보여줍니다.
- 각 추천 카드는 `레시피명`, `검색에 사용된 주재료`, `부족한 재료 수`, `추천 이유`를 보여줍니다.
- 장보기 부담이 너무 큰 레시피는 후순위로 밀어야 합니다.

### 4.3 장보기 목록

- 장보기 목록은 `선택한 레시피 1개 기준`으로 생성합니다.
- 장보기 목록에는 `냉장고에 없는 재료`만 단순하게 포함합니다.
- 장보기 목록은 `재료명`, `필요 수량`, `단위`를 표시합니다.
- 같은 재료가 레시피 내 여러 단계에서 반복되면 하나의 항목으로 합쳐서 보여줍니다.
- 사용자는 장보기 목록을 읽기만 하며, MVP에서는 체크박스나 구매 완료 상태는 제공하지 않습니다.

### 4.4 레시피 workflow

- 사용자는 추천 레시피를 선택한 뒤 `레시피 개요 -> 단계별 workflow -> 완료` 흐름으로 이동합니다.
- 레시피 개요 화면에는 `레시피명`, `예상 조리 시간`, `전체 재료`, `부족 재료`, `전체 단계 수`를 보여줍니다.
- workflow 화면은 한 번에 한 단계만 보여주는 구조로 구성합니다.
- 각 단계 화면에는 `단계 번호`, `설명`, `필요 재료`, `이전`, `다음` 버튼이 있어야 합니다.
- 마지막 단계에서는 `완료` 버튼을 보여줍니다.
- 사용자는 workflow 중 언제든 이전 단계로 돌아갈 수 있습니다.
- workflow 데이터는 검색된 레시피를 기반으로 생성하거나 적재한 `jsonl` 파일 형식으로 관리합니다.
- `jsonl`의 각 라인은 하나의 step을 의미해야 하며, 순서대로 읽어 workflow를 구성해야 합니다.
- 각 step은 최소 `recipeId`, `stepNumber`, `title`, `description`, `ingredients`, `estimatedMinutes` 필드를 가져야 합니다.
- MVP에서는 단계별 타이머, 음성, 이미지 확대, 수정 저장은 지원하지 않습니다.

## 5. 핵심 사용자 흐름

1. 사용자는 냉장고 입력창에 자연어로 재료를 입력합니다.
2. 시스템은 입력을 파싱해 냉장고 표를 생성합니다.
3. 사용자는 표에서 잘못된 재료 정보가 있으면 직접 수정합니다.
4. 냉장고 관리 agent는 파싱 결과를 고정 DB 스키마로 저장합니다.
5. 레시피 추천 agent는 냉장고 재료 중 주재료 5개를 선택해 첫 검색 plan을 만듭니다.
6. 검색 결과가 부족하면 주재료 수를 줄이며 단계적으로 검색 범위를 넓힙니다.
7. 시스템은 검색 결과를 정리해 추천 레시피 목록을 보여줍니다.
8. 사용자는 추천 레시피 하나를 선택합니다.
9. 시스템은 해당 레시피의 냉장고에 없는 재료만 장보기 목록으로 보여줍니다.
10. 시스템은 해당 레시피의 workflow용 jsonl step 데이터를 불러옵니다.
11. 사용자는 레시피 개요를 확인한 뒤 workflow를 시작합니다.
12. 사용자는 단계별 화면에서 이전과 다음을 누르며 조리를 진행합니다.
13. 마지막 단계에서 완료를 누르면 workflow가 종료됩니다.

## 6. 상세 기능 요구사항

### 6.1 냉장고 관리 화면

#### 사용자 목표

- 냉장고 속 재료를 빠르게 정리한다.
- 무엇이 있는지 표 형태로 명확하게 본다.

#### 필수 UI 요소

- 자연어 입력 텍스트 박스
- 입력 실행 버튼
- 파싱 결과 테이블
- 행 수정 기능
- 행 삭제 기능

#### 필수 동작

- 자연어 입력을 OpenRouter 기반 냉장고 관리 agent에 전달해야 합니다.
- agent 응답은 고정 스키마의 재료 행 배열이어야 합니다.
- 서버는 응답 스키마를 검증한 뒤 DB에 저장해야 합니다.
- 파싱 실패가 일부 발생해도 인식 가능한 항목은 먼저 표에 넣어야 합니다.
- 파싱이 모호한 항목은 사용자 수정 가능 상태로 표시해야 합니다.
- 남은 일수는 유통기한 기준으로 자동 계산해야 합니다.
- 표는 기본적으로 유통기한 임박순으로 정렬합니다.
- 사용자는 저장된 재료를 추가, 수정, 삭제할 수 있어야 합니다.

#### 빈 상태 / 오류 상태

- 입력 전에는 예시 문장을 보여줘야 합니다.
- 파싱 결과가 없으면 다시 입력하라는 안내를 보여줘야 합니다.
- 일부 항목만 파싱되면 실패한 텍스트를 사용자에게 알려줘야 합니다.
- OpenRouter 호출 실패 시 재시도 가능한 오류 상태를 보여줘야 합니다.

### 6.2 레시피 추천 화면

#### 사용자 목표

- 지금 냉장고 재료로 만들 수 있는 요리를 빠르게 찾는다.
- 먼저 써야 하는 재료를 우선 소비한다.

#### 필수 UI 요소

- 추천 레시피 카드 목록
- 추천 기준 설명 문구
- 각 카드의 부족 재료 수 표시

#### 필수 동작

- 냉장고 데이터가 있어야 추천을 실행합니다.
- 추천 agent는 냉장고 재료 중 검색용 주재료를 최대 5개 선택해야 합니다.
- 추천 agent는 검색 plan을 단계별로 생성해야 합니다.
- 1차 검색에서 결과가 부족하면 주재료 수를 줄여 다음 검색을 수행해야 합니다.
- 각 검색 단계의 입력과 결과를 로그로 남겨야 합니다.
- 레시피 데이터베이스의 각 레시피에 대해 보유 재료와 부족 재료를 계산합니다.
- 임박 재료 사용량이 높고 부족 재료가 적은 레시피를 상단에 정렬합니다.
- 각 카드에 추천 이유를 짧게 표시합니다.
- 추천 레시피 선택 시 장보기와 workflow로 이어지는 상세 흐름으로 이동합니다.

#### 빈 상태 / 오류 상태

- 냉장고 데이터가 없으면 먼저 재료 입력을 유도해야 합니다.
- 조건에 맞는 레시피가 없으면 추천 결과 없음 상태를 보여줘야 합니다.

### 6.3 장보기 목록 화면

#### 사용자 목표

- 선택한 레시피를 만들기 위해 꼭 사야 하는 것만 확인한다.

#### 필수 UI 요소

- 선택 레시피 이름
- 부족한 필수 재료 목록
- workflow 시작 버튼

#### 필수 동작

- 장보기 목록은 선택한 레시피 기준으로만 생성해야 합니다.
- 레시피 재료 중 냉장고에 없는 항목만 보여줘야 합니다.
- 재고가 일부 부족한 경우 부족한 수량만 계산해 보여줘야 합니다.
- 장보기 목록 확인 후 바로 workflow 시작이 가능해야 합니다.

#### 빈 상태 / 오류 상태

- 부족한 필수 재료가 없으면 `추가 구매 없이 조리 가능` 상태를 보여줘야 합니다.

### 6.4 레시피 workflow 화면

#### 사용자 목표

- 레시피를 순서대로 따라가며 조리한다.
- 현재 어느 단계인지 잃지 않는다.

#### 필수 UI 요소

- 레시피 개요 화면
- 단계 번호 표시
- 현재 단계 설명
- 이전 버튼
- 다음 버튼
- 완료 버튼

#### 필수 동작

- workflow 시작 전 레시피 개요를 먼저 보여줘야 합니다.
- 레시피 workflow는 레시피 DB에서 조회한 step 데이터 또는 해당 레시피용 jsonl 파일을 기준으로 구성해야 합니다.
- 단계는 순차적으로 전환되어야 합니다.
- 첫 단계에서는 이전 버튼을 비활성화합니다.
- 마지막 단계에서는 다음 버튼 대신 완료 버튼을 보여줘야 합니다.
- 완료 시 workflow 종료 상태를 명확히 보여줘야 합니다.

#### 빈 상태 / 오류 상태

- 단계 정보가 없는 레시피는 workflow 진입을 막아야 합니다.

## 7. 데이터 구조 결정

### 7.1 FridgeItem

- `id`: 재료 식별자
- `name`: 재료명
- `normalizedName`: 정규화 재료명
- `quantity`: 수량
- `unit`: 단위
- `expiryDate`: 유통기한
- `daysLeft`: 남은 일수
- `createdAt`: 생성 시각
- `updatedAt`: 수정 시각

### 7.2 Recipe

- `id`: 레시피 식별자
- `title`: 레시피명
- `ingredients`: 전체 재료 목록
- `requiredIngredients`: 필수 재료 목록
- `optionalIngredients`: 선택 재료 목록
- `steps`: 단계 목록
- `cookTime`: 예상 조리 시간

### 7.3 ShoppingItem

- `name`: 재료명
- `requiredQuantity`: 필요한 수량
- `unit`: 단위

### 7.4 WorkflowStep

- `recipeId`: 레시피 식별자
- `stepNumber`: 단계 번호
- `title`: 단계 제목
- `description`: 단계 설명
- `ingredients`: 해당 단계 사용 재료
- `estimatedMinutes`: 예상 소요 시간

### 7.5 RecipeSearchPlan

- `planId`: 검색 계획 식별자
- `recipeSearchAttempt`: 시도 순번
- `selectedIngredients`: 현재 검색에 사용한 주재료 목록
- `queryText`: 검색 질의
- `reason`: 이 조합을 선택한 이유
- `resultCount`: 검색 결과 수
- `nextAction`: 다음 검색 단계 설명

### 7.6 Workflow JSONL Format

- 파일 확장자는 `.jsonl`입니다.
- 한 줄은 하나의 workflow step을 의미합니다.
- 각 줄은 `recipeId`, `stepNumber`, `title`, `description`, `ingredients`, `estimatedMinutes`를 포함해야 합니다.
- stepNumber 기준 오름차순으로 읽어 workflow를 구성합니다.

## 8. 개발 우선순위

### P1

- OpenRouter 기반 냉장고 파싱 agent
- 고정 스키마 냉장고 DB 및 CRUD
- 검색 plan 기반 레시피 추천 agent
- 없는 재료 기준 장보기 목록 생성
- jsonl 기반 workflow 전환

### P2

- 파싱 정확도 개선
- 추천 랭킹 미세 조정
- 장보기 목록 UI 개선

## 9. 완료 기준

- 사용자가 자연어로 입력한 냉장고 재료가 표 형태로 정리되어 보여야 합니다.
- OpenRouter 응답 결과가 고정 DB 스키마로 저장되고 추가, 수정, 삭제가 가능해야 합니다.
- 유통기한 임박 재료를 우선 사용한 검색 plan이 실행되고, 결과 부족 시 검색 범위가 단계적으로 넓어져야 합니다.
- 선택한 레시피에 대해 냉장고에 없는 재료만 장보기 목록에 표시되어야 합니다.
- 사용자가 레시피 개요에서 workflow로 진입해 jsonl step 기준으로 단계별 이동과 완료를 할 수 있어야 합니다.

## 10. 개발 전 결정 필요 사항

### 10.1 OpenRouter 연동 결정

- 어떤 OpenRouter 모델을 냉장고 파싱 agent와 레시피 검색 plan agent에 사용할지 결정해야 합니다.
- 모델을 기능별로 분리할지, 하나의 모델로 통일할지 결정해야 합니다.
- 응답 형식을 JSON 스키마 강제 방식으로 받을지, 자유 텍스트 후 파싱할지 결정해야 합니다.
- 호출 실패, rate limit, timeout 시 재시도 정책을 몇 회까지 둘지 결정해야 합니다.

### 10.2 냉장고 DB 결정

- 실제 저장소를 SQLite로 둘지, Postgres로 둘지 결정해야 합니다.
- `normalizedName`을 어떤 기준으로 생성할지 결정해야 합니다.
- 같은 재료의 중복 입력을 언제까지 허용할지 결정해야 합니다.
- 유통기한이 없는 항목의 `daysLeft`를 `null`로 둘지, 별도 sentinel 값으로 둘지 결정해야 합니다.

### 10.3 냉장고 agent 결정

- 자연어 입력에서 날짜 표현을 어떤 포맷까지 허용할지 결정해야 합니다.
- 수량과 단위가 없는 입력에 기본값을 둘지 결정해야 합니다.
- 파싱 실패 항목을 부분 저장할지, 전체 실패로 돌릴지 결정해야 합니다.
- 사용자 수정 전 원문을 별도로 저장할지 결정해야 합니다.

### 10.4 레시피 검색 plan 결정

- 첫 검색에서 주재료 5개를 고를 때 임박일, 수량, 재료 중요도를 어떤 가중치로 계산할지 결정해야 합니다.
- 결과 부족의 기준을 `0건`, `3건 미만`, `5건 미만` 중 무엇으로 볼지 결정해야 합니다.
- 검색 범위를 줄이는 단계에서 1개씩 줄일지, 특정 규칙으로 재조합할지 결정해야 합니다.
- 여러 검색 단계 결과를 합칠 때 중복 레시피를 어떤 기준으로 정렬할지 결정해야 합니다.
- 검색 plan 생성은 모델이 전적으로 수행할지, 애플리케이션 규칙과 혼합할지 결정해야 합니다.

### 10.5 레시피 데이터베이스 결정

- 레시피 DB를 내부 seed 데이터로 시작할지, 외부 데이터 수집본으로 시작할지 결정해야 합니다.
- 재료 매칭을 문자열 일치로 할지, 정규화 사전 기반으로 할지 결정해야 합니다.
- 레시피 재료를 `필수`, `선택`으로 구분할지, MVP에서는 전체 재료만 둘지 결정해야 합니다.
- 검색 결과가 너무 많을 경우 최종 노출 개수를 몇 개로 고정할지 결정해야 합니다.

### 10.6 장보기 목록 결정

- 냉장고에 일부만 있는 재료를 부족 수량 계산 대상으로 볼지 결정해야 합니다.
- 장보기 목록에서 단위를 레시피 기준으로 유지할지, 냉장고 단위와 맞춰 변환할지 결정해야 합니다.
- `없는 재료` 판단에서 `quantity = 0`, `없음`, `단위 불일치`를 어떻게 처리할지 결정해야 합니다.

### 10.7 Workflow JSONL 결정

- workflow jsonl 파일을 레시피 저장 시 미리 생성할지, 조회 시 동적으로 생성할지 결정해야 합니다.
- step line에 이미지나 팁 필드를 포함할지 결정해야 합니다.
- jsonl 파일 저장 경로와 naming convention을 어떻게 둘지 결정해야 합니다.
- 레시피 step 데이터와 jsonl 파일 간 동기화 방식을 어떻게 유지할지 결정해야 합니다.

### 10.8 운영 및 관측 결정

- OpenRouter 호출 로그를 어디까지 저장할지 결정해야 합니다.
- 검색 plan 실행 로그를 디버그용으로만 저장할지, 영구 저장할지 결정해야 합니다.
- 추천 실패 원인을 사용자에게 어느 정도까지 노출할지 결정해야 합니다.
