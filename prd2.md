# 요리 여정 PRD v2

## 1. 문서 목적

이 문서는 현재까지 구현된 MVP 상태를 기준으로 다음 개발 단계를 정의하는 문서입니다.
기준은 아래 두 가지입니다.

- 현재 코드베이스에서 실제로 확인된 구현 상태
- [Archive/260316_README.md](/Users/yongsupyi/Desktop/frigo/Archive/260316_README.md)에 정리된 제품 비전

즉, `prd.md`가 1차 MVP의 구현 기준이었다면, `prd2.md`는 "현재 무엇이 되었고 무엇이 비어 있으며, 다음으로 무엇을 만들어야 하는가"를 정리하는 문서입니다.

## 2. 현재 구현 상태 요약

현재 프로젝트는 아래 4개 흐름이 동작하는 MVP 상태입니다.

- 냉장고 자연어 입력 및 재료 CRUD
- 레시피 추천 생성과 검색 plan 기록
- 레시피 기준 장보기 목록 생성
- 레시피 workflow 조회 및 단계 이동

현재 앱은 Docker Compose로 실행되며, 시작 시 migration, recipe seed 적재, workflow validation 후 FastAPI 서버가 올라갑니다.

## 3. 현재까지 개발된 사항

### 3.1 인프라 및 실행 환경

- [x] FastAPI + PostgreSQL + Docker Compose 기반 실행 구조가 있습니다.
- [x] 앱 컨테이너 시작 시 migration, recipe seed, workflow validation이 자동 수행됩니다.
- [x] OpenRouter 공통 클라이언트와 agent별 호출 구조가 분리되어 있습니다.
- [x] `.env` 기반 설정 구조가 있으며 `DATABASE_URL`, `OPENROUTER_API_KEY`, `OPENROUTER_BASE_URL`을 사용합니다.

### 3.2 냉장고 관리

- [x] 사용자는 자연어 텍스트로 냉장고 재료를 입력할 수 있습니다.
- [x] 냉장고 agent는 입력을 구조화하고 `fridge_items`, `fridge_input_logs`에 저장합니다.
- [x] 재료 목록 조회, 수정, 삭제가 가능합니다.
- [x] `days_left` 계산이 저장 로직과 수정 로직에 포함되어 있습니다.
- [x] OpenRouter가 없거나 실패할 경우를 위한 fallback 파싱 경로가 있습니다.

### 3.3 레시피 추천

- [x] 냉장고 재료를 바탕으로 검색 plan을 생성합니다.
- [x] 추천 로직은 `5 -> 4 -> 3` fallback 구조를 가집니다.
- [x] 결과가 많을 때 6개 조합으로 재검색하는 분기가 있습니다.
- [x] 각 추천 시도는 `recipe_search_plans`에 저장됩니다.
- [x] 현재 내부 레시피 DB는 raw 데이터 기반 10개 레시피로 구성되어 있습니다.
- [x] 추천 기준은 `search_keywords`와 냉장고 재료의 겹침 수입니다.
- [x] 대량 데이터 대응을 위해 `recipe_search_terms` 역색인 테이블과 SQL 후보 검색 구조가 반영되어 있습니다.

### 3.4 장보기 목록

- [x] 선택한 레시피와 냉장고 재고를 비교해 장보기 목록을 생성합니다.
- [x] 없는 재료, 부족 재료, 절반 이하 재고를 구분합니다.
- [x] 장보기 결과는 `shopping_list_runs`에 저장됩니다.

### 3.5 Workflow

- [x] 레시피별 workflow JSONL 파일이 존재합니다.
- [x] workflow는 `recipe_id`, `step_number`, `title`, `description`, `ingredients`, `tool`, `estimated_minutes` 구조를 가집니다.
- [x] workflow 검증 스크립트가 있습니다.
- [x] 사용자는 레시피 workflow를 한 단계씩 이동하며 볼 수 있습니다.

### 3.6 UI

- [x] 단일 대시보드 화면에서 냉장고 입력, 추천 실행, 장보기 확인이 가능합니다.
- [x] workflow 화면에서 이전, 다음, 완료 이동이 가능합니다.
- [x] MVP 수준의 템플릿 UI가 있습니다.

### 3.7 데이터 및 검증

- [x] 레시피 seed 10건과 workflow 10건이 정리되어 있습니다.
- [x] 데이터 무결성 테스트가 있습니다.
- [x] 추천 로직의 핵심 fallback/overlap 규칙 테스트가 있습니다.

## 4. 현재 부족한 사항

현재 구현은 `동작하는 MVP` 수준이며, [Archive/260316_README.md](/Users/yongsupyi/Desktop/frigo/Archive/260316_README.md) 기준의 제품 비전과는 아직 차이가 큽니다.

### 4.1 제품 기능 부족

- [ ] 홈 화면이 없습니다. 현재 `/`는 MVP 조작 화면이며, 요리 기록 대시보드가 아닙니다.
- [ ] 요리 잔디 심기 기능이 없습니다.
- [ ] 완료한 레시피 갤러리 기능이 없습니다.
- [ ] 사진 기반 추천 기능이 없습니다.
- [ ] 주간 식단 추천 기능이 없습니다.
- [ ] 주간 장보기 최적화 기능이 없습니다.
- [ ] 쿠킹 모드 전용 화면이 없습니다.
- [ ] 타이머, 단계 진행 기록, 실제 소요 시간 기록이 없습니다.
- [ ] 사용자가 레시피를 수정해 나만의 레시피로 저장하는 기능이 없습니다.
- [ ] 사용자 선호나 피드백에 따른 개인화 추천이 없습니다.

### 4.2 데이터 및 추천 품질 부족

- [ ] 내부 레시피 DB가 10건뿐이라 추천 다양성이 매우 낮습니다.
- [ ] 외부 raw 데이터에서 내부 DB로 정제하는 지속 가능한 파이프라인이 없습니다.
- [ ] 재료 동의어, 단위 변환, 재료 계층어 처리 로직이 약합니다.
- [ ] 추천은 현재 keyword overlap 중심이라 의미 기반 유사도나 다양성 제어가 없습니다.
- [ ] 장보기 목록에 가격 추정이 없습니다.

### 4.3 UX 부족

- [ ] LLM 응답이 스키마를 만족하지 않을 때 사용자가 수정 가능한 UI가 없습니다.
- [ ] 추천 결과, 장보기, workflow 흐름이 분리된 완성도 높은 사용자 경험으로 이어지지 않습니다.
- [ ] 모바일 중심 인터랙션, 스와이프, 큰 CTA, 진행 상태 시각화가 없습니다.

### 4.4 운영 및 품질 부족

- [ ] E2E 테스트가 부족합니다.
- [ ] OpenRouter timeout, retry, rate limit, 장애 대응이 없습니다.
- [ ] 운영 로그, 호출 로그, 추천 품질 로그가 없습니다.
- [ ] 사용자 계정, 다중 사용자 데이터 분리, 인증 체계가 없습니다.
- [ ] 성능 관점의 캐시 전략과 대량 데이터 벤치마크가 아직 부족합니다.

## 5. PRD v2 목표

PRD v2의 목표는 MVP를 "제품 형태가 보이는 베타" 수준으로 끌어올리는 것입니다.

이번 단계에서 집중할 목표는 아래 5가지입니다.

- 홈 경험 추가: 기록과 회고가 가능한 시작 화면 제공
- 추천 확장: 단일 추천에서 주간 계획까지 확장
- 요리 경험 강화: workflow 조회를 cooking mode로 확장
- 데이터 확장: 10건 seed를 넘어서는 내부 레시피 DB 구축 체계 마련
- 운영 안정화: 실패 대응, 로그, 테스트, 성능 측정 보강

- [ ] 추천, 장보기, cooking 완료 흐름에 대한 핵심 E2E 테스트가 존재합니다.
- [ ] 운영 환경에서 OpenRouter 실패가 발생해도 사용자 경험이 완전히 끊기지 않습니다.

## 11. 메모

- 현재 구현은 `MVP core flow`는 갖췄지만, archive 문서의 핵심인 `기록`, `주간 계획`, `쿠킹 모드`, `습관화`는 아직 미구현입니다.
- 따라서 다음 단계는 기능 수를 늘리는 것이 아니라, 이미 있는 추천/장보기/workflow를 사용자 여정 중심으로 재구성하는 것이 핵심입니다.
